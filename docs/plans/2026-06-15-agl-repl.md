# Plan: An interactive REPL for AgL (`agm repl`)

## Overview

AgL today only runs as a *whole program*: `agm exec FILE` (or `-c COMMAND`)
reads one complete source, and `WorkflowRuntime.run()` drives
`parse → resolve → typecheck → host-prep → eval` over it with **fresh** scope,
type, and value environments every call. There is no way to build up state
line-by-line, inspect bindings, ask "what type is this expression?", or probe a
workflow interactively.

This plan adds `agm repl`: an interactive read-eval-print loop for AgL with
**maximum interactivity, generality, and user-friendliness**. The hard part is
that AgL is statically analysed and its agent calls are real, costly,
side-effecting LLM/shell invocations — so the REPL must keep persistent session
state and evaluate each entry **exactly once**, never replaying prior agent
calls. Per the owner decision, we do this with a true **incremental persistent
environment**, not by re-running an accumulated program.

The interactive front end is built on **prompt_toolkit** (the library behind
ptpython / IPython), giving multiline editing, AgL-aware syntax highlighting,
tab-completion, and persistent history.

## Decisions

### Resolved with the owner

1. **State model — incremental persistent environment.** Each entry is parsed,
   resolved, type-checked, and evaluated **once** against a session environment
   that persists symbols, types, declarations, and runtime values across
   entries. Prior agent calls are **never** re-executed. (Rejected: pure replay
   and replay-with-memoization — both re-run or special-case side effects.)

2. **Agent backing — real runner, confirm-each-call by default; toggle to
   auto.** The REPL wires the same runner-backed agent as `agm exec` (from
   `[exec]` config / `--runner`). The **default** mode confirms before every
   live agent (or `exec` shell) call, showing the callee and the rendered prompt
   (option 3). A session setting / CLI flag switches to **auto** mode that fires
   calls immediately without prompting (option 2). The mode is selectable at
   launch and via the `:agent` meta-command.

3. **Front end — prompt_toolkit.** New dependency `prompt_toolkit>=3.0`. Used
   for multiline editing with an AgL-aware continuation predicate, syntax
   highlighting via a custom `Lexer` that wraps the existing AgL lexer,
   tab-completion (keywords / bindings / agent names / meta-commands), and
   persistent `FileHistory` under `~/.agm/`.

4. **`input` declarations — `:set` / pre-supplied only.** There is no `--input`
   file in an interactive session, so values for `input name: T` come from a
   `:set name=value` meta-command (and/or repeatable `--input KEY=VALUE` at
   launch). Evaluating an entry that needs an unset input is a clean error that
   points the user at `:set`. The REPL does **not** auto-prompt mid-eval.

### Minor decisions taken with recommended defaults (flag if you disagree)

5. **Redefinition is allowed and shadows.** Unlike whole-program mode (where a
   duplicate `let`/`record`/`input` is a static error), re-declaring a name in a
   later REPL entry **replaces/shadows** the prior session binding. This is the
   universal REPL convention. Mechanically we model the persistent session as an
   *implicit enclosing scope* of each entry, so a new declaration shadows without
   a duplicate-declaration error, then is promoted into the session on success.
   *(Alternative: error on redefinition like exec — rejected as hostile to
   interactive use.)*

6. **Result echoing.** A bare-expression entry (`ExprStmt`) prints its evaluated
   value using the default renderer (like Python's REPL echoing a value).
   A binding entry (`let`/`var`) prints `name : Type = value`. `print`, control
   flow, and declarations print only their own output / a terse confirmation.
   Echoing is on by default; `:set echo off` (or a `--quiet` launch flag)
   silences it. *(Flag if you'd prefer no auto-echo.)*

7. **Atomic entries.** An entry's new bindings/declarations are promoted into the
   session **only on full success**. A parse/resolve/typecheck failure changes
   nothing. A *runtime* exception reports the error and discards the entry's new
   bindings — but irreversible external side effects (an agent call or `exec`
   that already ran before the raise) cannot be undone; this is inherent and
   documented. *(Flag if you'd prefer partial promotion of pre-raise bindings.)*

8. **Meta-command prefix `:`.** Meta-commands use a leading colon
   (`:help`, `:reset`, …) so they never collide with AgL syntax (AgL has no
   statement beginning with `:`). Ctrl-D / `:quit` exits; Ctrl-C cancels the
   current entry (and interrupts a running agent call) without exiting.

9. **Static guarantees preserved per entry.** Every entry is fully type-checked
   against the session environment *before* evaluation — the REPL never weakens
   AgL's static checking; it only narrows the *unit* of checking from a whole
   file to one entry plus the accumulated session context.

10. **`WorkflowRuntime.run()` is left untouched.** The REPL needs incremental,
    per-entry execution with seeded environments, which `run()` deliberately does
    not offer. Rather than overload `run()`, the REPL gets a thin driver
    (`agm.agl.repl.session`) that calls the same pipeline passes directly with
    seed parameters. `agm exec` is unaffected.

## Library research

Search confirmed **prompt_toolkit** as the standard foundation for custom
Python REPLs, and the canonical way to build exactly this kind of tool:

- It powers **ptpython** and **IPython**, providing syntax highlighting,
  multiline editing, popup autocompletion, mouse support, and Emacs/Vi
  keybindings.
- The official **"Build an SQLite REPL"** tutorial is a near-exact template:
  `PromptSession` for the loop, a custom/`PygmentsLexer` for highlighting, a
  `Completer` for keywords, `FileHistory` for persistence, and a `Style` for
  colours.
- Highlighting: we will implement a custom `prompt_toolkit.lexer.Lexer` that
  drives the **existing AgL lexer** (`agm.agl.lexer.tokenize`) so highlighting
  matches the real grammar exactly — no second, drifting token definition. (A
  Pygments fallback lexer is the alternative if token→style mapping proves
  fiddly.)
- Completion uses a custom `Completer` fed from live session state (bound
  names, declared agents, keywords, meta-commands) rather than a static word
  list.
- History persists with `FileHistory(~/.agm/repl_history)`.

Sources:
- [prompt-toolkit/ptpython — a better Python REPL](https://github.com/prompt-toolkit/ptpython)
- [Tutorial: Build an SQLite REPL — prompt_toolkit docs](https://python-prompt-toolkit.readthedocs.io/en/stable/pages/tutorials/repl.html)
- [Building prompts — prompt_toolkit docs](https://python-prompt-toolkit.readthedocs.io/en/1.0.15/pages/building_prompts.html)

(Rejected: building on stdlib `readline`/`code.InteractiveConsole` — no real
highlighting, weaker multiline handling, more custom completion code for a worse
result.)

## Architecture

### The incremental session environment

The REPL models the **persistent session as an implicit enclosing scope** of
each entry. This is the key idea that makes incremental evaluation correct and
minimally invasive:

```
session env (persists across entries)
  ├─ symbol table   : name → binding (mutability, type)        [scope pass]
  ├─ type namespace : records / enums / type aliases           [typecheck pass]
  ├─ declarations   : agent decls, input decls                 [root-only]
  ├─ value scope    : name → runtime Value                     [eval]
  └─ inputs         : name → host value (from :set / --input)

per entry:
  parse(entry)                          → Program (entry only)
  resolve(entry, parent=session_scope)  → ResolvedProgram   (refs resolve into session)
  check(entry,  seed=session_typeenv)   → CheckedProgram    (full static check vs session)
  execute(entry, root=session_values)   → runs ONLY the entry's statements
  on success: promote new bindings/types/decls into the session env
```

Because we execute only the entry's statements (never the prior ones), agent
calls fire exactly once. Because resolve/check see the session as an enclosing
scope, references to earlier bindings resolve and type-check normally, and a new
declaration *shadows* an old one without a duplicate-declaration error.

### Changes to the firewalled passes (additive, optional params)

The AgL firewall (everything from the AST onward depends only on AST
dataclasses, never on Lark) is preserved — these are pure additive seam
parameters that default to today's behaviour:

- **`agm.agl.scope.resolve(program, *, parent_scope: ScopeNode | None = None)`**
  — when `parent_scope` is given, the entry's root `ScopeNode.parent` is set to
  it, so name lookups fall through to session bindings; new declarations live in
  the entry's own scope and shadow rather than collide. Default `None` → today's
  standalone behaviour (exec unchanged).

- **`agm.agl.typecheck.check(resolved, capabilities, *, seed_env: TypeEnvironment | None = None)`**
  — when `seed_env` is given, the checker pre-populates its `TypeEnvironment`
  with the session's record/enum/alias definitions and prior binding types
  (allowing override on redefinition). Default `None` → today's behaviour.

- **`agm.agl.eval`** — the interpreter already executes against a passed-in
  `Scope`. The REPL passes the persistent session `Scope` (or a child that is
  promoted on success). We add a small, well-tested helper for **redefinition**
  (replace an existing binding's value/mutability) since today's `Scope.define`
  assumes first-definition; alternatively the driver removes-then-defines. The
  interpreter walks only the entry program's `body`.

If seeding the checker's `TypeEnvironment` cleanly proves to need more than a
pre-population hook (e.g. the builder assumes an empty namespace), the fallback
is a `TypeEnvironment.fork(parent)` that consults a parent namespace on miss —
decided during M1 once the checker internals are confirmed. Either way the
change stays inside `typecheck/` behind the `seed_env` parameter.

### REPL host layer (`src/agm/agl/repl/`)

Kept independent of the CLI for generality (any host could embed it):

- **`session.py` — `ReplSession`**: owns the persistent env and the incremental
  driver. Public surface:
  - `eval_entry(text: str) -> EntryResult` — parse→resolve→check→eval one entry
    against the session, promote on success. `EntryResult` carries: kind
    (expression / binding / declaration / statement), echoed value + type,
    diagnostics, warnings, runtime error, trace path. This is the **pure,
    UI-free** core and is where the bulk of unit testing happens.
  - `type_of(text: str) -> str` — typecheck an expression and return its type
    without evaluating (`:type`).
  - `bindings() -> list[(name, Type, Value)]`, `agents()`, `inputs()`.
  - `set_input(name, raw)`, `reset()`, `load_file(path)`, `dump_source() -> str`.
- **`console.py` — prompt_toolkit integration**: `PromptSession` config, the
  `AglPromptLexer` (wraps `agm.agl.lexer.tokenize`), `AglCompleter`, the
  multiline continuation predicate, key bindings, history, and the styled
  banner/prompt (`agl> ` / continuation `...> `).
- **`meta.py` — meta-command dispatch**: parses a leading `:` line and routes to
  `ReplSession` / console actions; provides `:help`.
- **`render.py` — result formatting**: turns an `EntryResult` into styled output
  (reuses the runtime renderers for values).
- **`agents.py` — confirming agent wrapper**: wraps the runner-backed `AgentFn`
  to implement confirm-each-call vs auto modes and Ctrl-C cancellation.

### Multiline / incompleteness detection

AgL is indentation-based (the lexer injects INDENT/DEDENT). The continuation
predicate decides whether the buffer is a complete entry:

- Try `parse_program(buffer)`. If it succeeds → complete (Enter submits).
- If it fails with an "unexpected end of input / expected INDENT / unterminated"
  class of `AglSyntaxError` → incomplete (Enter inserts a newline, show `...>`).
- Any other syntax error → still submit (so the user sees the real error rather
  than being trapped). A trailing blank line force-submits regardless.

This reuses the real parser, so multiline behaviour tracks the grammar exactly
(blocks: `if`, `do`, `try`, `case`, `record`, `enum`).

### Meta-commands (v1 set)

| Command | Action |
|---|---|
| `:help` | List meta-commands and brief usage |
| `:quit` / `:exit` (or Ctrl-D) | Exit the REPL |
| `:reset` | Clear the entire session env (bindings, types, decls, inputs) |
| `:type EXPR` | Type-check `EXPR` against the session; print its type; no eval |
| `:bindings` / `:env` | List current bindings with types (and values) |
| `:agents` | List declared agents and their backing/mode |
| `:inputs` | List declared inputs and their current `:set` values |
| `:set name=value` | Supply a host input value (per decision 4) |
| `:set echo on|off` | Toggle result echoing (decision 6) |
| `:agent confirm|auto` | Switch agent-call mode (decision 2) |
| `:load FILE` | Parse+run an `.agl` file's statements into the session |
| `:save FILE` | Write the accumulated successful session source to a file |

### Agent-call confirmation flow (decision 2)

The default agent passed to the session is `runner_backed_agent_factory(...)`
(identical wiring to `agm exec`), wrapped by a confirming `AgentFn`:

- **confirm mode (default):** before dispatching, print `callee` + the rendered
  prompt (truncated, with a way to view full) and ask `[Y]es / [n]o / [a]lways`.
  `always` switches the session to auto for the rest of the session; `no` raises
  a catchable cancellation that surfaces as a clean REPL message (entry aborted).
- **auto mode:** dispatch immediately (matches `agm exec`).
- Ctrl-C during a live call interrupts the runner subprocess and aborts the
  entry without exiting the REPL.

Confirmation prompts happen synchronously inside eval; they use a minimal
prompt_toolkit confirmation (or `input()`), which is safe because the main loop
is idle while an entry evaluates.

### CLI command wiring

- `src/agm/commands/args.py`: add `ReplArgs` (slots dataclass):
  `inputs: list[str]`, `strict_json: bool | None`, `max_iters: int | None`,
  `runner: str | None`, `auto_agents: bool`, `quiet: bool`,
  `no_log: bool`, `log_file: str | None`.
- `src/agm/commands/repl.py`: `run(args: ReplArgs)` — load `[exec]` config
  (reused; the REPL shares exec's runner/agents/timeout settings), build the
  `WorkflowRuntime`-style agent backing, construct `ReplSession` + console, run
  the loop. `--input` pairs pre-seed `:set` values. Trace logging via the same
  `resolve_log_file` helper as exec.
- `src/agm/cli.py`: `@app.command(name="repl")` mirroring the `exec_cmd`
  pattern (options above; `_help`/`_dry_run` handled consistently — `--dry-run`
  in a REPL means "type-check only, never eval", a useful exploration mode).
- `src/agm/completion.py`: arg-value completions for `repl` (e.g.
  `complete_path_argument` for `--log-file`) if any.

This honours the project rule that `commands/` mirrors the CLI command tree:
`repl` is a single top-level command → `commands/repl.py` (no subgroup).

## Testing strategy (TDD, 100% coverage)

Per the project rule, every level gets tests and we keep 100% coverage of
`src/`. The architecture is deliberately split so the hard logic is testable
without a terminal.

- **Unit — `tests/test_agl_repl_session.py`** (the bulk): drive `ReplSession`
  directly with strings. Cover: persistence across entries; redefinition/
  shadowing; expression echoing; `:type` without eval; atomic-on-error
  (failed entry leaves session unchanged); agent calls fire exactly once
  (assert with a counting fake `AgentFn`); `:set` input flow incl. unset-input
  error; `:reset`, `:load`, `:save`; warnings surfaced per entry. Multi-scenario:
  several distinct input/agent-response combinations, not one happy path.
- **Pass-seam unit tests**: extend `tests/test_agl_scope.py` and
  `tests/test_agl_typecheck.py` for the new `parent_scope` / `seed_env`
  parameters (default-None behaviour unchanged; seeded behaviour resolves and
  shadows correctly).
- **Console — `tests/test_agl_repl_console.py`**: use prompt_toolkit's
  `create_pipe_input()` + `DummyOutput()` to feed scripted keystrokes headlessly.
  Cover multiline continuation (incomplete `if`/`do` keeps prompting), submit on
  completion, Ctrl-D exit, Ctrl-C cancel, completer suggestions, and the lexer
  producing tokens for a sample line.
- **Command — `tests/test_repl_command.py`**: `ReplArgs` wiring, config
  resolution, `--input` pre-seed, `--dry-run` → typecheck-only, mutually
  exclusive `--no-log`/`--log-file`, and CLI registration (don't assert exact
  help text, per the rules).
- **Agent confirmation — in session/console tests**: confirm/auto modes,
  `always` switching mode, `no` aborting the entry, with a fake agent.

Coverage of `console.py` is reached via pipe-input tests; any genuinely
untestable terminal glue is minimised and, if unavoidable, raised with the owner
rather than `# pragma: no cover`'d unilaterally.

## Risks & mitigations

- **mypy `disallow_any_expr` vs prompt_toolkit.** The strict config forbids
  `Any`-typed expressions. prompt_toolkit ships `py.typed` but some APIs may
  surface `Any`. Mitigation: confine prompt_toolkit usage to `console.py`/
  `agents.py` behind thin, fully-annotated wrappers; add focused shims under
  `stubs/` (the repo already uses `stubs/` + `MYPYPATH=src:stubs`) if specific
  symbols type as `Any`. **No `type: ignore`** without owner approval (per
  CLAUDE.md). This is the single most likely source of friction; budget for it
  in M2.
- **Seeding the checker's `TypeEnvironment`.** If pre-population isn't enough,
  fall back to a `fork(parent)` lookup (kept inside `typecheck/`). Confirmed in
  M1 against the real checker internals.
- **Nested prompt_toolkit (confirmation inside eval).** Use a minimal confirm
  prompt; verified by headless tests.

1. **M1 — Incremental core (no UI).** Add `parent_scope`/`seed_env` seams +
   `Scope` redefinition helper; build `ReplSession.eval_entry` and the
   persistent env with promote-on-success; `:type`. Full unit + seam tests.
   *(Highest risk; do first, strict TDD.)* Commit.
2. **M2 — prompt_toolkit console.** `console.py` (PromptSession, `AglPromptLexer`,
   `AglCompleter`, multiline predicate, history), `agm repl` command + CLI +
   `ReplArgs`. Resolve the mypy/`Any` story here. Headless console + command

3. **M3 — Meta-commands + echoing.** `meta.py`, `render.py`, all `:` commands,
   result echo + `:set echo`. Tests. Commit.
4. **M4 — Agent backing.** Confirming wrapper (confirm/auto, `:agent`,
   `--auto-agents`), `:set` input flow, `--input` pre-seed, trace logging via
   `resolve_log_file`, Ctrl-C cancellation. Tests with fakes. Commit.
5. **M5 — Docs & polish.** README mention, `docs/commands.md` full reference,
   `agm repl --help` text, banner/UX polish. `just check` green. Commit.
