# Plan: AgL type casts (`as`) and convertibility test (`as?`)

## Overview

AgL needs an explicit type-conversion operator and a convertibility predicate:

```agl
let s: text = 1 as text          # int → text
let j: json = 1 as json          # int → json
let n: int  = userInput as int   # text → int (strict parse; may fail at runtime)

if userInput as? int             # convertibility test → bool
  let n = userInput as int
  print n
else
  print "not an int: ${userInput}"
```

AgL is statically typed, so the set of *possible* conversions is fixed and
known at type-check time: a disallowed source/target pair is a **static error**,
checked before the program runs. A conversion pair that is statically allowed
but cannot be proven to succeed (parsing, validation, numeric narrowing) is a
**runtime-fallible** cast that raises a new catchable `CastError` on failure.
The `as?` operator runs the same conversion as a trial and reports success as a
`bool` without raising.

This work touches the full AgL pipeline (lexer → grammar → AST → scope →
typecheck → runtime/eval) plus the language reference docs and the e2e suite.

## Resolved Owner Decisions

These were resolved with the owner and frame the implementation.

| # | Decision |
|---|----------|
| D1 | **Broad conversion scope, including parsing.** `as text` (render any renderable value) and `as json` (canonicalize any JSON-shaped value) are *total*; plus `text → {bool,int,decimal,json,list,dict,record,enum}` parsing, `json → {…}` validation/extraction, and `decimal → int` narrowing — these are *runtime-fallible*. |
| D2 | **Cast failures raise a new built-in `CastError`** exception with structured fields (`message`, `trace_id`, `source_type`, `target_type`, `raw`). |
| D3 | **Strict parsing for casts** — a deliberate, cast-scoped flip from the emphatic lenient-JSON default. `text as T` requires the text to be exactly one well-formed JSON value (no `json-repair`, no fence/prose recovery), then jsonschema-validates into `T`. Agent/`exec` output parsing stays lenient. |
| D4 | **Non-integral numeric narrowing fails.** `3.5 as int` raises `CastError`; `3.0 as int` succeeds as `3`. Consistent with the strict story and the existing wire rule (a non-integral JSON number does not satisfy `int`). |
| D5 | **`as` binds tight (Rust-like):** looser than unary `-`, tighter than `* /`. So `-1 as text` = `(-1) as text`, and `1 + 2 as text` = `1 + (2 as text)` (a type error — `(1+2) as text` must be parenthesized). |
| D6 | **Redundant casts are allowed as no-ops.** If the source type is already assignable to the target (identity, `int as decimal`, JSON-shaped `as json`), the cast type-checks and applies any existing coercion at runtime. |
| D7 | **Convertibility test is spelled `EXPR as? T`** and yields `bool`. (`bool` because AgL has no optional/nullable/union types.) |
| D8 | **`as?` mirrors `as`'s static rules exactly.** `x as? T` type-checks iff `x as T` type-checks; a statically-impossible pair is the same static error. For statically-total casts the result is statically always `true` (allowed, per D6). |

### D1 — The conversion matrix

`as` is defined over the **data** types only. `unit`, `agent`, and function
types never participate as source or target (they are not renderable, not
JSON-shaped, and not parseable) — naming them on either side is a static error.

The matrix, by target type:

| Target `T`            | Allowed source `S`                                                                 | Kind |
|-----------------------|-------------------------------------------------------------------------------------|------|
| `text`                | any renderable type: `text, json, bool, int, decimal, list, dict, record, enum`     | **total** (render via `render_value`) |
| `json`                | any JSON-shaped type: `text, json, bool, int, decimal, list[..], dict[..]`          | **total** (canonicalize via `value_to_json_obj`) |
| `bool`                | `bool` (no-op); `text`/`json` (value must be a JSON bool)                            | fallible (except no-op) |
| `int`                 | `int` (no-op); `decimal` (must be integral); `text`/`json` (must be an integral number) | fallible (except no-op) |
| `decimal`             | `decimal`/`int` (no-op/widen); `text`/`json` (must be a number)                     | fallible (except no-op/widen) |
| `list[E]`, `dict[text,V]` | identical container type (no-op); `text`/`json` (parse/validate into the shape) | fallible (except no-op) |
| record / enum         | identical nominal type (no-op); `text`/`json` (parse/validate against the schema)   | fallible (except no-op) |

Everything not listed is a **static cast error**, e.g. `bool as int`
(booleans never coerce to/from numbers — existing language rule), `1 as agent`,
`someFunction as text`, `record as json` (records/enums are **not** JSON-shaped;
see *Open points* below).

Note `int as text`, `json as text`, `bool as text` are the *total* `→ text`
cases; `int as json` / `1 as json` is the *total* `→ json` case. These cover the
owner's two motivating examples and never raise.

### D2 — `CastError`

Add a built-in catchable exception, registered like the existing
`IndexError`/`KeyError`/`MatchError` family in `agm.agl.typecheck.types`:

```
exception CastError
  message:     text
  trace_id:    text
  source_type: text   (* display name of the value's static type, e.g. "text" *)
  target_type: text   (* display name of the cast target, e.g. "int" *)
  raw:         text    (* rendered source value, for diagnostics *)
```

It is catchable like any built-in exception (`try … catch CastError as e …`),
non-constructible only if we follow the abstract-base pattern — here it is a
concrete catchable type. Its name joins `BUILTIN_EXCEPTION_NAMES` (cannot be
redeclared as a user record/enum/alias).

### D3 — Strict parsing (the flip)

`text as T` parses the source text with a **strict** JSON parser
(`json.loads`-equivalent: the text must be exactly one well-formed JSON value,
leading/trailing whitespace aside) and then validates the parsed object against
`derive_schema(T)`. This is intentionally **different** from agent/`exec` output
parsing, which keeps the lenient `json-repair` recovery default. The divergence
is deliberate: a cast operates on ordinary in-program text, where silent repair
of malformed input (`"42 apples" as int` → `42`) would be surprising.

`json as T` skips the text-parse step entirely and validates the existing
`json` value into `T`.

The codec already contains the exact seams to reuse (see *Runtime core*):
`JsonCodec._parse_strict` (strict `json.loads`), `derive_schema` +
`Draft202012Validator`, and `_json_to_value(obj, typ)` (typed-`Value`
construction with integral-decimal normalization). The plan factors these into a
shared module so the lenient agent path and the strict cast path do not
duplicate logic.

### D4 — `decimal as int` and non-integral numbers

`decimal as int` succeeds iff the decimal has no fractional part
(`d == d.to_integral_value()`), yielding the exact integer; otherwise it raises
`CastError`. The same integrality rule governs `text`/`json` → `int` (a JSON
number with a non-integral value does not satisfy `int`), reusing the existing
`_normalize_integral_decimals` behavior in the codec.

### D5 — Precedence

Insert a new **cast** precedence level between `unary` and `multiplicative`:

```
postfix > juxt > unary(-) > [CAST] > * / > + - > comparison > not > and > or
```

Grammar sketch (left-associative; covers both `as` and `as?`):

```lark
?multiplicative: multiplicative (STAR|SLASH) cast | cast
?cast: cast KW_AS type_expr        -> cast_expr
     | cast AS_QUESTION type_expr  -> cast_test
     | unary
```

Consequences: `-1 as text` = `(-1) as text`; `2 * 3 as text` =
`2 * (3 as text)`; `1 + 2 as text` = `1 + (2 as text)`; `f x as int` =
`(f x) as int` (juxtaposition is tighter); chaining `x as json as text` =
`(x as json) as text`. `a as? int and b` = `(a as? int) and b`.

### D6 — Redundant casts

When the source's static type `is_assignable` to the target, the cast
type-checks and is a runtime no-op modulo existing coercions: `int as decimal`
widens, `X as json` canonicalizes a JSON-shaped value into `json`
representation, and exact-type casts return the value unchanged. No warning is
emitted (consistent with the language's other no-op-friendly constructs).

### D7 / D8 — `as?` convertibility test

`EXPR as? T` is a single token-pair operator at the cast precedence level.
It type-checks under the **same** matrix as `EXPR as T` (D8) and always has
type `bool`. At runtime it performs the identical conversion as a trial:
returns `true` if it would succeed, `false` if it would raise `CastError`.
For statically-total casts (`x as? text`, redundant pairs) it is statically
`true` but still legal.

## AST design

Use a **single** node with a discriminator rather than two near-identical
nodes, to avoid duplicating grammar/transform/typecheck/eval scaffolding (the
only differences are the result type and the raise-vs-return-bool wrapper):

```python
@dataclass(frozen=True, slots=True)
class Cast:
    """A type cast (`expr as T`) or convertibility test (`expr as? T`)."""
    expr: Expr
    target_type: TypeExpr
    test_only: bool          # False → `as` (yields T); True → `as?` (yields bool)
    span: SourceSpan = dc_field(compare=False)
    node_id: int = dc_field(compare=False)
```

`Cast` joins the `Expr` union in `agm.agl.syntax.nodes`. The resolved semantic
target `Type` is attached to a typecheck **side table keyed by `node_id`** (per
the side-table convention — never mutate frozen nodes), and read by the
evaluator.

## Implementation milestones

Each milestone ends green on `just check` and is committed on its own.

### M1 — Lexer, grammar, AST (components 1–3)

- **Lexer** (`agm.agl.lexer`): emit a distinct `AS_QUESTION` token when `?`
  immediately follows the `as` keyword (adjacency, mirroring the existing
  `INDEX_LSQB` adjacent-bracket trick). `as` with a following space stays
  `KW_AS`; `as ?` (spaced) is not the test operator. Add lexer tests.
- **Grammar** (`grammar/agl.lark`): add the `cast` precedence level (D5) with
  `cast_expr` and `cast_test` productions over a `type_expr` right-hand side.
- **AST** (`agm.agl.syntax.nodes`): add the `Cast` node; extend the `Expr`
  union. `agm.agl.parser.transform`: add `cast_expr`/`cast_test` handlers
  building `Cast(test_only=…)` with span + node id.
- Tests: `test_agl_lexer.py`, `test_agl_parser.py` (precedence/associativity
  cases from D5, both operators), `test_agl_ast.py`. Parse-rejection fixtures
  under `tests/agl/rejections/parse/` (e.g. `as` with a missing type).

### M2 — Scope pass (component 4)

- `agm.agl.scope`: traverse `Cast.expr` so the inner expression resolves; the
  `target_type` is a `TypeExpr` resolved during typecheck (consistent with how
  other type annotations are handled). No new bindings.
- Tests: `test_agl_scope.py` (cast sub-expression resolves; undefined var
  inside a cast is a scope error).

### M3 — Typecheck (component 5)

- Register `CastError` in `BUILTIN_EXCEPTIONS` / `BUILTIN_EXCEPTION_NAMES`
  (`agm.agl.typecheck.types`).
- Add `_check_cast` in `agm.agl.typecheck.checker`:
  1. infer the source type;
  2. resolve `target_type` to a semantic `Type`;
  3. validate the pair against the matrix (D1) — central helper
     `cast_classification(source, target) -> CastKind` returning one of
     `TOTAL_NOOP | TOTAL_RENDER | TOTAL_JSON | FALLIBLE | STATIC_ERROR`;
  4. on `STATIC_ERROR`, raise `AglTypeError` with a clear message naming both
     types;
  5. store the resolved target `Type` (and `CastKind`) in the `CheckedProgram`
     side table keyed by `node_id`;
  6. result type = `BoolType()` when `test_only`, else the resolved target.
- The matrix logic builds on existing `is_assignable` / `is_json_shaped` /
  `render`-ability predicates (extend `typecheck.types` with a small
  `is_renderable`/`is_castable` helper rather than duplicating membership
  checks).
- Tests: `test_agl_typecheck.py` (result types for `as`/`as?` across the
  matrix); type-rejection fixtures under `tests/agl/rejections/type/`
  (`cast_bool_to_int`, `cast_record_to_json`, `cast_agent`, `cast_to_unit`,
  `cast_function`, etc.).

### M4 — Runtime conversion core (component: runtime)

- Factor the reusable conversion seam out of `agm.agl.runtime.codec` into a
  shared helper (new `agm.agl.runtime.convert` or existing `serialize`/`codec`
  module) so both the lenient agent path and the strict cast path share it
  (avoid duplication):
  - `parse_json_strict(text) -> object` (strict `json.loads`);
  - `json_obj_to_value(obj, type) -> Value` with `derive_schema` +
    `Draft202012Validator` validation and integral-decimal normalization
    (lifted from `_json_to_value` / `_normalize_integral_decimals`).
- Implement the single conversion entry point
  `convert_value(value, source_type, target_type) -> Value` realizing the
  matrix:
  - `→ text`: `render_value`;
  - `→ json`: `JsonValue(value_to_json_obj(value))`;
  - `text → T`: `json_obj_to_value(parse_json_strict(text), T)`;
  - `json → T`: `json_obj_to_value(value.raw, T)`;
  - `decimal → int`: integrality check (D4);
  - no-op/assignable cases (D6): return the value, applying int→decimal widening
    / json canonicalization.
  Conversion failures raise a Python-level error carrying enough context to
  build the `CastError` value (source/target display names, rendered raw).
- Tests: `test_agl_runtime.py` for the shared helper; confirm `JsonCodec` still
  passes after the refactor (no behavior change to the lenient path).

### M5 — Eval (component 6)

- `agm.agl.eval.interpreter._eval_cast`: read the resolved target `Type` and
  `CastKind` from the side table, evaluate `expr`, then:
  - `test_only=False`: call `convert_value`; on failure raise an `AglRaise`
    wrapping a `CastError` `ExceptionValue` (fields per D2);
  - `test_only=True`: run `convert_value` under a trial; return
    `BoolValue(True)` on success, `BoolValue(False)` on the conversion failure
    (only the conversion failure — not unrelated runtime errors).
- Tests: `test_agl_eval.py` for each matrix cell, success and `CastError`
  paths, and `as?` true/false.

### M6 — Docs + e2e

- **Reference docs** (`docs/agl/reference/`, language-only, no implementation):
  - `types.md` — new "Casts and convertibility" section: the matrix, total vs
    fallible, the strict-parsing note, `decimal → int` integrality, redundant
    no-op casts; extend the assignability section to cross-reference.
  - `expressions.md` — `as` / `as?` operators and their precedence (D5).
  - `exceptions.md` — add `CastError` to the built-in exception table.
  - `lexical-structure.md` / `grammar.md` — `as?` token and grammar rules.
- **Architecture doc** (`docs/arch/agl.md`): note the `Cast` node, the
  cast side table, the shared strict-parse/validate conversion helper, and that
  cast parsing is strict while agent/exec parsing stays lenient.
- **e2e programs** (`tests/agl/programs/casts/`): multi-scenario `.agl` +
  `.scenarios.json` exercising casts in combination with other features —
  `as text`/`as json` rendering, `text`→scalar parsing (success + `CastError`),
  `json`→record/enum validation, `decimal as int` (integral + non-integral),
  `as?` branching, casts inside arithmetic/interpolation/`let`, and
  `try/catch CastError`. Cover both successful conversions and caught failures.
- Confirm `just check` green and 100% `src/` + command coverage.

## Open points to confirm during implementation

- **`record as json` stays a static error.** This follows the existing language
  invariant that records/enums/exceptions are *not* JSON-shaped and must be
  rendered to text before entering a `json` slot (`is_json_shaped`,
  `types.md`). The broad-scope decision (D1) does **not** override this; if the
  owner wants `record as json` to perform structural JSON encoding, that is a
  separate change to the JSON-shape model and should be raised explicitly.
- **`as?` on a statically-total cast** is always `true` (D8); we allow it
  without a warning. Flag if a lint-style warning is later desired.

## Risks

- **Grammar conflicts.** The new `cast` level sits between `unary` and
  `multiplicative` and reuses `KW_AS`. `as` is already consumed by
  `catch_pattern` in a different grammar context, so no LALR conflict is
  expected, but the parser tests must lock in precedence/associativity.
- **Lenient/strict split.** The refactor in M4 must not change the agent/`exec`
  lenient path; the existing codec tests are the guardrail.
- **Side-table discipline.** The resolved target `Type` must flow via the
  `node_id` side table; do not mutate frozen `Cast` nodes.
